# 게시글 페이징 학습 인터뷰 결과 리포트

## 1. 인터뷰 개요
- **주제**: 게시글 목록보기 페이징 기술의 원리 파악 및 단계별 학습 설계
- **목표**: 라이브러리 없이 밑바닥부터 페이징의 핵심 로직(DB 쿼리, 페이지 계산, UI 구현)을 스스로 구현하며 학습한다.
- **일시**: 2026년 3월 20일

---

## 2. 인터뷰 문답 요약 (Q&A)

**Q1: 페이징을 구현할 때 가장 궁금하거나 파헤쳐보고 싶은 핵심 지점은 무엇인가요?**
- **A**: 처음 접하는 기술이라 원리부터 차근차근, 라이브러리 없이 기본기부터 단계를 나눠서 배우고 싶습니다.

**Q2: 100만 개의 데이터를 나눠서 보여주어야 하는 기술적인 이유는 무엇일까요?**
- **A**: 성능 면에서 전체를 불러오면 시간이 오래 걸리지만, 나눠서 가져오면 시간과 메모리를 절약할 수 있습니다. 또한 사용자 입장에서도 한꺼번에 너무 많은 데이터가 보이면 보기 싫어집니다.

**Q3: 특정 페이지(예: 3페이지)를 요청할 때 DB에 전달해야 할 핵심 숫자 정보는?**
- **A**: 몇 개의 게시글을 보여줄지 정하고, 그 숫자에 맞춰 이전 페이지들만큼을 제외(Skip)한 다음 데이터를 가져와야 합니다.

**Q4: '건너뛸 개수(Offset)'를 구하는 공식을 일반화한다면?**
- **A**: `(현재 페이지 번호 - 1) * 한 페이지당 게시글 수` 만큼을 건너뛰어야 합니다.

**Q5: 하단 페이지 버튼을 그리기 위해 DB에 추가로 물어봐야 할 정보는?**
- **A**: 전체 페이지 수를 계산하기 위해 전체 게시글의 개수(`COUNT`)를 알아야 합니다.

**Q6: 전체 105개 글, 페이지당 10개씩일 때 총 페이지 수는?**
- **A**: 11개입니다. 10개보다 크면 다음 페이지로 넘어가야 합니다.

**Q7: '나머지가 있으면 1을 더한다'는 논리를 구현하는 선호 방식은?**
- **A**: 정수 연산(나머지 연산자 `%`)과 삼항 연산자를 활용하는 방식이 직관적입니다.

**Q8: 페이지 버튼이 너무 많을 때(예: 100개) 버튼 블록을 계산하는 기준은?**
- **A**: 현재 페이지를 기준으로 앞뒤로 몇 개를 보여줄지(예: 현재 페이지 +/- 2) 결정해야 합니다.

**Q9: 버튼 숫자가 1보다 작아지거나 전체 페이지보다 커지는 경우의 처리는?**
- **A**: `if` 조건문을 사용하여 시작 페이지는 1보다 작을 수 없고, 끝 페이지는 전체 페이지 수보다 클 수 없도록 방어 코드를 작성합니다.

**Q10: 사용자가 클릭한 페이지 번호는 서버의 어디에서 가장 먼저 받아야 하나요?**
- **A**: 사용자의 요청을 가장 먼저 마주하는 `Controller`에서 받아야 합니다.

## 3. 최종 학습 로드맵 (5단계)

### [1단계] DB 데이터 추출 (Offset & Limit)
- 순수 SQL `LIMIT ?, ?` 사용법 익히기
- Java에서 Offset 계산 로직 구현

### [2단계] 전체 데이터 수 및 총 페이지 계산
- `COUNT(*)` 쿼리 수행 및 Java에서의 페이지 수 올림 처리
- `TotalPage = (TotalCount / Size) + (TotalCount % Size > 0 ? 1 : 0)`

### [3단계] 기본 페이징 버튼 UI (Mustache)
- Mustache의 반복문을 활용한 단순 숫자 버튼 나열
- 클릭 시 `?page=n` 형태의 GET 요청 전송

### [4단계] 페이지 블록(묶음) 처리
- 현재 페이지 중심의 버튼 노출 범위 계산
- `[이전]`, `[다음]` 버튼 활성화 조건 로직 작성

### [5단계] 예외 상황 방어 (Boundary Check)
- 존재하지 않는 페이지 요청 시 처리
- 첫 페이지와 마지막 페이지에서의 버튼 노출 제어
