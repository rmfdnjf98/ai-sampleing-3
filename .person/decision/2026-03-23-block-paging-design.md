# 실무형 블록 페이징(Block Paging) 설계

- 날짜: 2026-03-23
- 참여: 사용자 + Gemini CLI

## 배경
기존의 단순 '이전/다음' 방식에서 벗어나, 사용자가 여러 페이지를 한 번에 보고 이동할 수 있도록 숫자를 묶음(Block) 단위로 제공하는 UI를 구축하고자 함.

## 핵심 논의 및 결론

### 1. 블록 단위 설정
- **`blockSize = 5`**: 한 번에 5개의 페이지 번호씩 화면에 노출한다. (예: 1~5, 6~10, ...)

### 2. 핵심 변수 및 계산 공식
- **`startPage`**: 현재 블록의 첫 번째 페이지 번호.
  - 공식: `((currentPage - 1) / blockSize) * blockSize + 1`
- **`endPage`**: 현재 블록의 마지막 페이지 번호.
  - 공식: `startPage + blockSize - 1`
  - **안전장치**: `Math.min(endPage, totalPages)`를 적용하여 존재하지 않는 유령 페이지 노출을 방지한다.

### 3. 이전/다음 블록 이동
- **`prev`**: 이전 블록의 마지막 페이지로 이동.
- **`next`**: 다음 블록의 첫 번째 페이지로 이동.

## 💡 사용자 질문 & 학습 포인트 (Q&A)

### Q1. `Math.min(endPage, totalPages)`은 왜 쓰는 건가요?
**A. "유령 페이지 방지용 안전장치"입니다.**
- **비유**: 5장씩 꽂는 사진 앨범이 있는데, 실제 사진은 8장뿐인 상황.
- **계산**: 공식대로라면 2번째 블록은 6, 7, 8, 9, 10번을 보여줘야 함.
- **결과**: `Math.min(10, 8)`을 통해 실제 사진이 있는 **8번**까지만 보여주고, 존재하지 않는 9, 10번(유령 페이지)은 잘라냄.

### Q2. 블록 페이징(B 방식)이 왜 더 복잡한가요?
**A. "현재 내가 몇 번째 묶음에 있는지"를 계속 계산해야 하기 때문입니다.**
- 단순히 숫자를 나열하는 게 아니라, `startPage`와 `endPage`라는 동적인 경계선을 계산하고, 이를 다시 `totalPages`와 비교하는 수학적 그룹핑 로직이 들어가기 때문입니다.
